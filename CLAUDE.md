# CLAUDE.md — Ours (private couples app)

Context for anyone (human or model) picking up this codebase.

## What this is

**Ours** is a private, just-for-two relationship app. One Expo codebase runs on iOS, Android, and web. Original brief in `Details.md`; it has since grown: a public landing page, a post-login home dashboard, a notification service, a calendar-based memory log, heart reactions, a shared bucket list, optional pairing, and (phase 2) a daily prompt with mutual reveal, time capsules, a date planner, per-partner wishlists with secret gift plans, and Sunday weekly reflections. **No stubs, no mock states.**

## Stack

| Layer | Tech |
|---|---|
| App | Expo SDK 53 + React Native 0.79 + Expo Router 5 (`react-native-web` for web) |
| API | ONE Vercel serverless function (`api/index.ts`) routing to modules in `api/_routes/` |
| Database | CockroachDB (Postgres wire protocol, `pg` driver) |
| Realtime | Ably (server publishes via REST, clients subscribe via scoped tokens) |
| Auth | Email+password, `node:crypto` scrypt, `jsonwebtoken` JWTs (30-day) |

## Commands

```sh
npm install
npm run migrate        # applies db/schema.sql (idempotent); RE-RUN after schema changes
npm run typecheck      # tsc --noEmit over app/ src/ api/ scripts/
npm run build:web      # expo export --platform web -> dist/
npx vercel dev         # API on :3000 (local)
npx expo start         # app; press w for web
```

Environment: `.env` from `.env.example` (`DATABASE_URL`, `ABLY_API_KEY`, `JWT_SECRET`, `EXPO_PUBLIC_API_URL` (stays EMPTY on Vercel), `VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY`/`VAPID_SUBJECT` for Web Push (generated once via `npx tsx scripts/generate-vapid.ts`), `MASTER_ENCRYPTION_KEY` for envelope encryption at rest (256-bit, base64; generated once via `npx tsx scripts/generate-master-key.ts`; if unset the app runs with encryption OFF so nothing breaks), `ADMIN_PASSWORD` gating `/admin/dashboard` (unset = dashboard locked), and `CRON_SECRET` gating the scheduled reminder endpoint `/api/cron/reminders` that the GitHub Actions in `.github/workflows/` call (unset = reminders return 503; the Actions also need repo secrets `CRON_SECRET` (same value) + `APP_URL`)). Deployed via GitHub import into Vercel; env vars live in Vercel project settings.

`npm test` runs vitest over `api/**/*.test.ts` (server route unit tests; `memory-item.test.ts` guards the per-user hearts rule).

## Product structure

- **Landing** `app/(auth)/welcome.tsx`: public marketing page (hero, three feature cards, CTA). Signed-out users at `/` land here.
- **Auth** sign-up/sign-in. **Signup auto-creates a solo space** (couple row); pairing is optional.
- **Pairing** `app/pair.tsx`: NOT a gate. Share your code or join theirs; joining migrates everything you authored into their space (see `api/_routes/couple-join.ts`).
- **Home** `app/(tabs)/index.tsx`: days-together count (basis: earliest anniversary milestone, else couple created_at; when there is NO anniversary yet the hero shows "Here is to you two ♥" + an inline "add the day it began" link instead of a misleading "0 days"), hero marks (both partners' avatars with a ♥ between; falls back to the initials monogram), invite banner when solo, "Your journey" points/level card, resurfaced memory (1 year ago today > 1 month ago today > random older), "Coming up soon" milestones (always rendered, with a "+ Add" header link; empty-state copy when none), latest pinned note, resurfaced memory. The bucket list is NO LONGER on Home (moved into the Wishes tab). Powered by ONE aggregate call: `GET /api/home` (still returns `bucket` for now, unused by the client).
- **Wishes** `app/(tabs)/wishlist.tsx` (route path stays `/wishlist`; tab LABEL is "Wishes"): the merged home for the bucket list AND both wishlists, chosen over the old separate "Our list" screen (deleted `app/(tabs)/list.tsx`) + Wishlist tab. Top segmented control: **Ours** / **Mine** / **Theirs**. *Ours* = the shared bucket list (`bucket_items`), each item tagged Experience or Item (category, v12); add/rename (tap title)/complete/delete, either partner edits; completed items STAY, struck-through under "Done together" with a "Done {date}" (completed_at, v12). *Mine* = your own wishlist. *Theirs* = your partner's wishlist, where the gift composer offers **secret** (hidden from them forever, the old default) OR **visible/on-their-behalf** (`secret:false`, they see it). Both wishlist and bucket titles carry a category (experience|item); wishlist title/url/notes stay encrypted, bucket + category are plaintext. `GET /api/bucket` returns all (done ASC, completed_at DESC); `PATCH /api/bucket/:id` takes `{ done }` (stamps/clears completed_at) OR `{ title }` OR `{ category }`; bucket routes now publish `bucket.updated`. `GET /api/wishlist` + POST carry `category`.
- **Memories** `app/(tabs)/memories.tsx`: two parts. (1) Month calendar; days that hold a memory render a ♥ instead of the number; tapping a (non-future) day opens the composer pinned to that date (`memory_date` column). (2) Photo timeline with heart reactions and a full-photo viewer. Tapping the photo or note opens the viewer; the MessageCircle button (always visible) expands the comment thread INLINE under the card, Facebook-style (`MemoryComments` with `variant="light"`; the dark variant stays in the viewer). On web >= 900px calendar and timeline sit side by side. `MemoryImage` falls back to fetching the full photo for rows created before thumbnails existed (thumb_data null but has_photo true); do not remove that fallback.
- **Notes** `app/(tabs)/notes.tsx`, the letter wall (redesigned): composer card at TOP (serif input whose placeholder is a rotating "spark" starter; ✦ Sparkles button deals another, Smile toggles the emoji palette, Lock seals a capsule, wax-seal ♥ disc sends; no helper copy under it, just the LockBadge). Wall cards carry a tiny deterministic tilt (`tiltFor(id)`, pinned cards sit straight on blush), and ONE footer row: avatar + name + relative time on the left, icon tools on the right (Pin icon, gold + filled when pinned; Trash2 on your own; then the heart). Hearts: you can heart only your partner's notes; your own shows a bare "♥ Loved" once they do (`note_hearts` v9, PATCH /api/notes/:id { hearted }, settles on the server response). Kept deliberately word-light per user feedback: icons over labels, no explanatory text.
- **Avatars ("marks")** `src/components/Avatar.tsx`: curated set of lucide icons on tinted discs (ids must match `AVATAR_IDS` in `api/_routes/auth-profile.ts`); no upload, no emoji. Picked in Settings → Profile ("Your mark"), stored on `users.avatar` (v9), carried ONLY by `/api/auth/me` (self + partner) — list routes never join it because every author in a couple is one of the two `useAuth()` people. Fallback = serif initial disc. Shown on the Home hero, note cards, comment threads, and the Settings partner row.
- **Partner nicknames** (`users.partner_nickname` v11): a pet name you give your partner. Stored on the VIEWER's own row ("the name I call my partner"), so each partner sets their own independently. Resolved at `/api/auth/me`: `partner.display_name` becomes `nickname || realName`, so every screen reading `useAuth().partner` shows it with no other change; `partner.realName` + `partner.nickname` ride along for Settings and the notification-text swap. Set in Settings → Your space (calls `updateProfile({ partnerNickname })` then `refresh()`). The two server-generated name surfaces are handled explicitly: the notifications screen swaps `realName`→`nickname` in each stored line (`withNickname`), and `home.ts` resolves the pinned-note author. Plaintext by design (a term of endearment). Nullable/blank clears it. Catch-guarded selects (auth-me + home) keep a pre-v11 deploy from failing.
- **Milestones** yearly recurrence for anniversary/birthday, second-level countdowns.
- **Daily prompt** (`api/_routes/prompts.ts`, home card + `app/(tabs)/prompts.tsx` history): one static-pool question per day, deterministic by date; answers are private until BOTH partners submit, then reveal. One submission per person per day.
- **Streak** (streak math in `api/_lib/streak.ts`, wired in `prompts.ts`): a day counts when BOTH partners answered that day's prompt. **Computed from the source of truth** (`computeStreak(mutualDays, todayUTC)`): the whole streak is re-derived every read from the set of days both partners answered, so it can never drift (the old incremental counter double-fired / mis-timezoned and showed wrong numbers). Pure + unit-tested (`api/_lib/streak.test.ts`, 10 cases incl. "2 days reads as 2"). UTC days, one grace day per Monday-to-Sunday UTC week. `couples.current_streak_days/longest_streak_days/last_streak_date` are now just a cache for admin analytics, refreshed on each reveal by `advanceStreak`; `grace_used_week` is unused. `streakStateFor` = read view. Surfaced from day one: hero chip on Home, prompt-card lines, and the prompts screen header now carries a persistent "how the streak works" explainer (user could not tell what it counted). **Streak celebration**: when a day becomes mutual (both answered), the sanctioned hearts shower fires once per prompt day as positive reinforcement (Home `celebrateStreak`, guarded by `celebratedDateRef`; fires from the answer-submit response for the second answerer and from the `prompt.revealed` event for the first). No leaderboards.
- **Relationship points + levels** (in `home.ts`, was "story chapters"): the second retention hook, **reframed from a book (pages/chapters) to points/levels** because the book metaphor lacked context (user: "I have no idea what it means, why, what it gives"). Everything durable the couple builds earns points (memories x5, dates x5 but ONLY once the date has happened (`dates_done`: accepted AND proposed_for null-or-past), prompt answers x3, bucket done x3, notes x2, milestones x2, comments x1); `LEVELS` thresholds name the level (First Glance -> Ever After, 10 of them; the three book-specific names were replaced). Computed on read from count(*) subqueries in the home aggregate (raw `counts` ride along), no counter column, cannot drift. Response shape: `story: { points, level, levelTitle, levelStart, nextAt, counts }`. Home "Your journey" card: TAP expands an explainer + the point-source breakdown inline, "All levels" opens the level-map Sheet (which leads with a plain-language explanation). Reaching a level shows a one-time sealed "A new level" ceremony with a top-3 `pointsSummary`; last celebrated level in `localStorage['ours.story-chapter']` (key kept; web only, native skips). LEVELS + POINT_SOURCES are duplicated client-side in `app/(tabs)/index.tsx`; keep both in sync with home.ts.
- **Chat** (`app/chat.tsx`, `api/_routes/messages.ts`, v10): a direct partner-to-partner thread. Reached via a round **chat button parked above the add FAB** (`src/components/ChatButton.tsx`, rendered once in the tabs layout; shown only when paired + on the main tab routes; hides while the add menu is open via the `src/lib/fabMenu.ts` shared signal; carries an unread dot). Bodies are **encrypted at rest** (envelope, `messages.body_ct`). Live over Ably (`message.created`, plaintext body over the TLS + scoped channel, like notes/memories) + a best-effort generic Web Push to the away partner. Chat deliberately writes **no notification rows** (would flood the bell); unread is tracked by `users.chat_seen_at` (`GET /api/messages/unread` for the badge, `POST /api/messages/seen` to clear). Inverted FlatList, optimistic send, `?before=` pagination. **Media (v15)**: a message can carry a photo (`image_thumb` ~480px shown inline + in the list, `image_data` full fetched on tap via `GET /api/messages/:id`; plaintext base64 like memory photos, NOT encrypted). The thumb rides the `message.created` event (list-weight, deliberate minor deviation from "no images over Ably"). An image bubble has an **"Add to timeline"** button (`POST /api/messages/:id { action:'to-timeline' }` server-side copies the photo into a `memories` row, which DOES notify). **Last seen**: `/messages/seen` publishes `chat.seen`; the sender sees a "Seen" receipt under their newest message once the partner's `chat_seen_at` passes it; GET returns `partnerSeenAt`.
- **Hearts shower** `src/components/HeartsRain.tsx` (mounted once in the tabs layout next to NudgeToast): a ~2.5s rain of ♥ glyphs (oxblood/gold/blush, staggered falls, pointerEvents none). Fires live on the `nudge` Ably event (not your own), on app open via `showHearts()` when `/api/home` returns `nudged: true` (an unseen nudge from the last 48h, computed with a FILTER clause on the unseen-notifications query; Home guards it to once per open), and now on a **streak extension** (a prompt day becoming mutual, once per day). It is the ONE sanctioned shower; reuse `showHearts()` for celebrations, do not add a second confetti system.
- **Time capsules**: notes and memories accept `sealedUntil`. Partner-authored sealed rows come back with content stripped server-side (never client-side). After the reveal date they show as "ready to open"; first open is recorded in `capsule_opened_at` and notifies the author. Memory opens happen via GET /api/memories/:id; note opens via PATCH /api/notes/:id { open: true }.
- **Date planner** `app/(tabs)/dates.tsx` (rebuilt): three segments **Upcoming** (accepted, not-yet-happened) / **Proposals** (open, waiting on you/them) / **Past** (happened or resolved). Flow: propose title/location + optional date + optional **time** (HH:MM, `proposed_time` v13) → the OTHER partner accepts/declines/counters (PATCH /api/dates/:id { action }, counter carries time too); accepting a dated proposal still creates a custom milestone in the same transaction. Confirmed dates land in Upcoming; **reminders** fire ~24h/6h/1h before via the cron (see Reminders). Once a date has passed, its Past row invites **"Log how it went"** (`action:'complete'`): a `ReflectSheet` with a 1-5 **star rating**, a **note**, an optional **photo that becomes a real timeline memory** (the client POSTs /api/memories then passes `memoryId`), and a **"save as a date idea"** toggle (writes `date_ideas`, encrypted title/location). `completed_at`/`rating`/`reflection`(_ct)/`memory_id` on the proposal (v13). GET /api/dates also returns `ideas`; the propose sheet's **"Surprise them"** picks a random idea from the couple's saved `date_ideas` ∪ a built-in list (`BUILT_IN_IDEAS`). Every row still opens a status-aware `ProposalDetail`. Date notifications + reminders are deliberately generic (title is encrypted).
- **Wishlist** `app/(tabs)/wishlist.tsx`: each partner owns a list, the other reads it. `secret` rows are gift plans added to the PARTNER's list, hidden from the owner forever (server-side filter, do not weaken). Partner toggles `gotten`; owner never sees who got it.
- **Weekly reflection / week-in-review** (`api/_routes/reflection.ts`): Monday-to-Sunday UTC counts computed on read, home card on Sundays only, savable keepsakes browsable at `app/(tabs)/reflections.tsx`. `computeReflection` now also gathers a **snapshot**: a few of the week's memory thumbnails with their (decrypted) notes (`gallery`) + a couple of decrypted note excerpts (`noteHighlights`); the Home card and history render a little photo strip + a note quote above the counts. Saving **freezes** the snapshot into `weekly_reflections.snapshot` (JSONB, v14) so the keepsake survives later edits (ON CONFLICT now upserts the snapshot). NOTE: this also fixed a latent bug where the highlight note was returned as raw (empty under encryption) instead of decrypted.
- **Notifications** `app/(tabs)/notifications.tsx` (hidden tab, reached via bell): every partner action (kinds: nudge, memory, note, milestone, partner, bucket, prompt, capsule, date, wishlist), stored in `notifications` AND pushed live over Ably. Unread dot in `src/lib/notifications.tsx`; `users.notifications_seen_at` tracks read state.
- **Navigation**: 5 tabs (Home, Memories, Notes, Dates, Wishes); Milestones, Settings, Notifications, Prompts, Reflections are hidden tabs (`href: null`) reached from home/bell/TopNav. (The old `list` hidden tab was deleted when the bucket list moved into Wishes.) Web >= 900px gets `src/components/TopNav.tsx`; native/narrow gets bottom tabs + `src/components/HeaderActions.tsx`. Home has no header; its hero row carries Nudge/Bell/Settings. Bottom-right controls, stacked: the chat button sits above the wax-seal add FAB.
- **Admin dashboard** `app/admin/dashboard.tsx` (route `/admin/dashboard`, OUTSIDE the auth/tabs groups): counts-only analytics. Overview (couple/user counts, paired vs solo, encrypted couples, active-this-week), streaks, 30-day signups chart, **content magnitude bars incl. chat messages**, a **30-day activity chart with a Daily/Cumulative toggle** (`activity` series, all content types), and a **per-couple breakdown** (`couples`: short opaque id `left(id,8)`, member count, total items + a per-type line + joined/last-active dates, sorted by volume). Gated by one shared password in the `ADMIN_PASSWORD` env var: `POST /api/admin/auth` returns a 12h admin JWT (`{ admin: true }`, no `sub`, kept separate from user auth in `api/_lib/admin.ts`), `GET /api/admin/stats` requires it. Reads only COUNTS + timestamps (per-couple counts were added at the owner's request); never reads couple content, never decrypts. The admin token lives only in the screen's state, never localStorage, never the shared `api()` client. If `ADMIN_PASSWORD` is unset the dashboard stays locked (503).

- **Scheduled reminders** (`api/_routes/cron-reminders.ts`, `.github/workflows/`): a PWA cannot schedule its own notifications, so scheduled GitHub Actions ping `POST /api/cron/reminders?kind=...` (gated by `CRON_SECRET`, Bearer; unset = 503). `kind=daily` (once/day, `daily-reminder.yml`): for every couple that added NO memory and NO note that UTC day, both partners get a "keep one small thing" Web Push. `kind=dates` (hourly, `date-reminders.yml`): for each accepted, not-yet-logged, dated proposal, both partners get a **generic** (title is encrypted) push at ~24h/6h/1h before, one per threshold, tracked by `reminded_24/6/1` flags (pure decision logic in `api/_lib/date-reminders.ts`, unit-tested). Neither writes notification rows (self-reminders must not crowd the bell). Setup needs `CRON_SECRET` in Vercel + repo secrets `CRON_SECRET` (same) & `APP_URL`; the workflows are UTC, so tune the cron times to your evening.

## Design system implementation (phase 2 foundation)

- `src/theme.ts` owns everything: `sp` spacing scale (4..56, only those values), `radius` (sm 6 photos/inputs, md 10 cards, lg 16 sheets, pill), `text` presets (display/title/subtitle/section/body/bodySerif/caption/micro; Fraunces = couple-authored content, system sans = chrome), `motion` (press 0.98/120ms, fade 180ms, sheet spring 220/26), semantic colors (`surface`, `surfaceRaised`, `surfaceSealed` oxblood, `onSealed`, `ink/inkMuted/inkFaint`, `hairline`, `accent` gold, `positive` olive). Legacy aliases (cream/rose/blush...) remain for stragglers; prefer semantic names.
- **Theme presets (P1.5, shipped web-only)**: 5 palettes (`parchment` default, `dusk`, `meadow`, `tide`, `petal`) built by `makeColors(seed)` in `src/theme.ts`; spacing/radius/type/motion never vary. Because every screen bakes `colors` into module-scope StyleSheets, the preset id is read SYNCHRONOUSLY from `localStorage['ours.theme']` at bundle evaluation; switching (Settings → Appearance, web-gated) = `persistThemePreset(id)` + `updateProfile({ themePreset })` + one `location.reload()`. The look is SHARED per couple: `couples.theme_preset` (v8; the v7 `users.theme_preset` is abandoned, kept additive-only), either partner sets it, and the other partner picks it up on their next app load via `refresh()` in `src/lib/auth.tsx` (reloads once when the couple's preset differs from the evaluated one). An inline script in `app/+html.tsx` paints `localStorage['ours.theme-bg']` before the bundle loads so non-parchment themes don't flash. Native has no sync storage at module scope, so it always runs parchment and hides the picker; a live ThemeProvider refactor is the eventual native path. New preset = add to `PALETTES` + `THEME_PRESETS` in theme.ts AND `THEME_PRESET_IDS` in `api/_routes/auth-profile.ts`.
- `src/components/kit.tsx` is the component library: Screen, Section, Card (+ `sealed` variant), PressableCard, AppPressable (shared press scale), PrimaryButton/SecondaryButton/IconButton, TextField (bottom-hairline style, gold focus), Pill, ListRow, Empty (✦ + serif line), Skeleton (olive 8%, 400ms delay), ErrorState, FormError, FadeIn. `src/components/Sheet.tsx` = bottom sheet native / right panel on web >= 900. Old `src/components/ui.tsx` was deleted; do not recreate it.
- Icons: `lucide-react-native` everywhere (works on web via react-native-svg), stroke 1.75. No emoji as chrome; ♥ ✦ stay as content marks.
- Haptics via `src/lib/haptics.ts` (`tapHaptic` on tab/segment changes, `successHaptic` on submit/heart/accept/unseal), web no-op.
- No shadows anywhere; depth = hairline border + surfaceRaised.

## Performance contract (why the app is fast; do not regress)

1. **List endpoints NEVER return full-resolution photos.** `GET /api/memories` carries only ~15 KB thumbnails (`thumb_data`, client-generated at 360px/q0.55). The full photo (1200px/q0.7) is stored in `photo_data` and fetched only by `GET /api/memories/:id` when the viewer opens.
2. **Home is one request** (`/api/home`), all queries in `Promise.all`.
3. Ably events carry ids/metadata, never image payloads.
4. Remaining latency is Vercel cold start + DB round trip: keep the Vercel function region next to the CockroachDB region (project Settings -> Functions).

## API surface (all via api/index.ts router; add endpoint = module in _routes/ + one table entry)

auth/signup (creates solo space) · auth/login · auth/me (partner name resolved to nickname) · auth/profile (name/notifications/pushToken/themePreset/avatar/partnerNickname) · auth/account · couple (GET) · couple/create · couple/join (migrates content) · memories (GET list/POST) · memories/:id (GET full photo / PATCH heart / DELETE any couple memory) · notes (GET/POST, list carries hearts + hearted_by_me) · notes/:id (PATCH pin or heart or open / DELETE own) · milestones (GET/POST) · milestones/:id (DELETE) · notifications (GET list+unseen / POST mark seen) · bucket (GET/POST) · bucket/:id (PATCH done|title|category / DELETE) · home (GET aggregate) · messages (GET list / POST send text+/-photo) · messages/seen (POST advance chat read cursor + publish chat.seen) · messages/unread (GET badge count) · messages/:id (GET full image / POST { action:'to-timeline' }) · push/subscribe (POST, store Web Push subscription in push_token) · push/vapid-public-key (GET, no auth) · nudge (POST) · ably-token (GET) · admin/auth (POST password -> admin JWT) · admin/stats (GET counts + per-couple analytics, admin JWT) · cron/reminders (POST/GET, CRON_SECRET-gated, ?kind=daily|dates; called only by GitHub Actions)

`_lib/`: db (pg pool), auth (scrypt/JWT, `requirePairedUser` lazily creates a space for legacy accounts), ably, notify (insert notification + publish + best-effort Web Push to the partner), notification-routes (kind -> deep-link path), invite (code + space creation, wraps a fresh DEK), push (real Web Push via `web-push` + VAPID; native APNs/FCM still needs store creds), **envelope** (encryption at rest, see below), respond (route wrapper: CORS/methods/errors).

### Envelope encryption at rest (`api/_lib/envelope.ts`)

The single encrypt/decrypt boundary (a future move to E2E is a swap of this module + a client layer, not a route rewrite). Two key layers: a **master key** (256-bit, base64 in `MASTER_ENCRYPTION_KEY`, never in the DB) wraps a per-couple **DEK** (256-bit random, stored AES-256-GCM-wrapped on `couples.wrapped_dek`); field values are encrypted with the couple's DEK. Ciphertext layout per value: `iv(12) || ciphertext || authTag(16)` as one `BYTEA`.

- API: `encryptionEnabled()`, `freshWrappedDek()` (used by `invite.createCoupleForUser`), `encryptField(coupleId, plaintext) -> Buffer|null`, `decryptField(coupleId, blob) -> string|null`, `readField(coupleId, ct, plaintext)` (prefer decrypted ct, else the plaintext column).
- **Graceful degradation**: with no master key set, every helper returns null and callers read/write plaintext exactly as before, so the deployed app keeps working until the key is provisioned. Legacy couples get a DEK minted lazily on first use.
- **Storage pattern**: each encrypted column has a `<field>_ct BYTEA` beside the original. When encryption is ON, writes store the plaintext column empty (`''`/NULL) and the ciphertext in `_ct`; reads use `readField` (decrypt `_ct`, fall back to plaintext for old/never-backfilled rows). Old plaintext columns are NOT dropped (additive only). `scripts/backfill-encryption.ts` encrypts pre-existing rows (idempotent; blanks plaintext after).
- **Encrypted fields**: `memories.note`, `love_notes.body` (covers note/memory time-capsule content), `daily_prompt_answers.text`, `wishlist_items.title/url/notes`, `date_proposals.title/location`, `messages.body` (chat). NOT encrypted (by design): milestone titles (list-view), bucket titles, reflection counts, timestamps, FKs, wishlist `secret` flag. `notify()` text must never embed an encrypted value (it would leak into the plaintext `notifications` table) — wishlist/date notifications are deliberately generic.
- **Pairing migration**: `couple-join.ts` re-wraps the migrated encrypted fields (`memories.note_ct`, `love_notes.body_ct`) from the joining user's DEK to the target couple's DEK inside the merge transaction (`recryptBlob`), so content stays readable under the new key. Any encrypted field that becomes migratable later must be added to that route's `MIGRATED_CT` list.
- **Ably** carries plaintext for these fields (server already holds it; the channel is TLS + subscribe-only scoped tokens). "Encryption at rest" is about the database, not the realtime hop.
- **`/api/auth/me`** returns `encryption: boolean` plus `encryptionCode` (the "seal code": `keyFingerprint()` = SHA-256 of the couple's DEK with a fixed context string, mapped to 8 chars of the invite alphabet as `XXXX-XXXX`; one-way, identical for both partners, shown in Settings → Privacy so the couple can compare phones). The client threads both through `useAuth()`; `LockBadge` on composers and the Privacy copy never overclaim when the key is absent.
- **Key rotation (not automated)**: to rotate `MASTER_ENCRYPTION_KEY`, for every couple: unwrap `wrapped_dek` with the OLD master key, re-wrap with the NEW one, write it back (one pass, DEKs and field ciphertext are untouched). Rotating a couple's DEK instead means decrypting every field with the old DEK and re-encrypting with the new one. Never change the master key without re-wrapping, or all data becomes unreadable.

## Data model

`couples` (+ `wrapped_dek BYTEA` for envelope encryption, streak counters v6 now cache-only, `theme_preset` v8 shared look) · `users` (couple_id, notifications_enabled, notifications_seen_at, push_token, `avatar` v9 mark id, `chat_seen_at` v10 chat read cursor, `partner_nickname` v11 pet name for the partner) · `memories` (photo_data full, thumb_data small, memory_date DATE, note, `note_ct`) · `love_notes` (pinned, `body_ct`) · `milestones` (kind: anniversary|birthday|custom) · `notifications` (couple_id, actor_id, kind, text) · `memory_hearts` (memory_id+user_id PK) · `note_hearts` (note_id+user_id PK, v9) · `bucket_items` (title, done) · `daily_prompt_answers` (`text_ct`) · `wishlist_items` (`title_ct`/`url_ct`/`notes_ct`) · `date_proposals` (`title_ct`/`location_ct`) · `messages` (v10 chat: couple_id, sender_id, body, `body_ct`; + v15 `image_thumb`/`image_data` plaintext base64). Newer additive columns/tables: **v12** `bucket_items.category`(experience|item) + `.completed_at`, `wishlist_items.category`; **v13** `date_proposals.proposed_time`/`rating`/`reflection`(_ct)/`memory_id`/`completed_at`/`reminded_24|6|1`, new `date_ideas` (couple's saved date-idea pool, title/location encrypted); **v14** `weekly_reflections.snapshot` (JSONB frozen week snapshot). The `_ct` columns are v4 envelope-encryption ciphertext (see envelope section); plaintext columns remain for fallback and are not dropped. Schema is idempotent (`CREATE TABLE IF NOT EXISTS` + `ADD COLUMN IF NOT EXISTS`); always safe to re-run migrate.

## Realtime events (channel `couple:{coupleId}`)

`note.created` `note.pinned` `note.deleted` `note.hearted` (id + count + by) `memory.created` (id only, clients refetch) `memory.deleted` `memory.hearted` `nudge` `partner.joined` `notification` (feeds the bell dot) `message.created` (chat: full message incl. plaintext body + `image_thumb` if any; the ChatButton dot + open chat both listen) `chat.seen` (partner advanced their read cursor -> "Seen" receipt) `bucket.updated` (Ours list changed -> Wishes tab refetches). Clients ignore their own events by actor/author id (except where noted).

## Security invariants (do not weaken)

1. Every couple-scoped query filters by the authenticated user's `couple_id`; privacy lives in the API.
2. JWT carries only `sub`; `requireUser` re-reads the user row per request so couple_id is never stale.
3. Ably tokens are subscribe-only, scoped to the user's own couple channel (`/api/ably-token`); the API key never leaves the server.
4. Spaces cap at 2 members; joining a full space is rejected. Deletes are own-content-only EXCEPT memories, which either partner may delete (couple_id still enforced; intentional, confirmed by the user).

## Design system (non-negotiable)

Direction: **aged love letters**. Parchment `#F4ECDD` ground, espresso ink `#33241C`, oxblood `#7E382C` primary (wax seal), ochre gold `#B8862F` flourishes, dry olive `#77743F` secondary. Deliberately NOT the blush/rose default of AI-generated apps. All tokens in `src/theme.ts` only, including `onRose` for text on oxblood; never hardcode hex in screens.

- Fraunces serif for display/headings and memory/note bodies; system sans elsewhere.
- 1px hairline borders, no shadows; pressed states darken, never bounce.
- Content columns max ~560-680px, centered.
- Emoji as micro-touches only (♥ nudge, ✦ pins, calendar hearts); no mascots, no confetti.
- **Copy rules: warm second person, and NEVER use em dashes (—) in user-facing copy.** Use commas or periods.

## Honest limitations (do not fake)

- Web Push (browser / installed PWA) is REAL: VAPID keys + `web-push` in `api/_lib/push.ts`, subscription JSON stored in `users.push_token`, delivered on every `notify()`. Needs the three `VAPID_*` env vars set on Vercel. Closed-app push to a real NATIVE iOS/Android binary still needs APNs/FCM store credentials; those are not provisioned. In-app realtime via Ably is real.
- Billing: Settings shows a free state; no payment flow by design.

## Gotchas

- **Never add a non-underscore file directly under `api/`**; Vercel Hobby caps deployments at 12 functions. `api/index.ts` must remain the only one; the `/api/:path*` rewrite in `vercel.json` routes to it (a `[...path].ts` catch-all was tried and 404'd in production; keep the rewrite pattern). Rewrite order matters: the api rewrite must precede the SPA fallback.
- **PWA / iOS standalone**: web output is `"static"` (app.json) so Expo Router honors `app/+html.tsx`, where the manifest link + `apple-mobile-web-app-*` meta tags live. Static assets in `public/` (manifest.json, sw.js, favicon.png, icons/) copy to `dist/` root verbatim and are served before the SPA rewrite (proven: `_expo/` JS loads fine). Icons are real PNGs generated dependency-free by `scripts/generate-icons.ts` (no sharp/canvas). `vercel.json` `headers` set the manifest content-type + sw.js no-cache. The service worker is registered client-side from `src/lib/push-web.ts` (web only, no auto-prompt); permission is requested only from the Settings toggle.
- `jsonwebtoken` not `jose` (jose is ESM-only; this package is CJS).
- `expo-asset` must stay installed (expo-font's web loader needs it).
- Component `style` props are `StyleProp<ViewStyle>`, not `ViewStyle`.
- Confirms are inline two-step UI (Alert.alert does not work on web).
- Milestone dates are validated `YYYY-MM-DD` text fields (no native picker dependency).
- Invite codes use `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (no 0/O/1/I).
- Windows machine; PowerShell 5.1 has no `&&`; use the Bash tool for chained commands.
- iOS Safari auto-inflates small text ("text size adjust"), which blew the 11px tab labels past the fixed-height tab bar and clipped them. `app/+html.tsx` sets `html { -webkit-text-size-adjust: 100% }`; the tab bar also pins `lineHeight: 14` + `tabBarAllowFontScaling: false`. Keep all three.

## Hard constraints from the user

- **Never touch git.** The user commits/pushes/deploys themselves.
- After schema changes, remind the user to run `npm run migrate` against the production `DATABASE_URL`. Symptoms of a missed migration: home stuck on its error state, uploads failing, 500s in Vercel function logs mentioning missing columns or relations. Data is never lost by this; the API just errors until the migration runs.

## Universal add button (`src/components/AddMenu.tsx`)

Wax-seal FAB anchored bottom-right (offset above the tab bar + safe-area bottom), rendered once in `app/(tabs)/_layout.tsx`. Tapping raises a VERTICAL column of four labelled actions (Add memory, Add note, Add wishlist item, Propose a date; labels to the LEFT of each disc so nothing can overlap — the original quarter-arc collided its own labels and was replaced). Milestone adds live on the Home/Milestones screens. Staggered fade+rise, no bounce; dismiss on tap-outside or swipe-down (scrim `PanResponder`). Visible only on the 5 main tab routes (`usePathname` gate); composers/viewers are RN `Modal`s that portal above it. Each action `router.navigate`s with a fresh `compose` nonce; the target screen opens its composer via `useComposeParam` (`src/lib/useComposeParam.ts`) — wishlist always opens on the user's own list.

## Safe-area bottom (`src/lib/safeArea.ts`)

`useSafeBottom()` is the ONLY correct way to read the bottom inset. react-native-safe-area-context's web provider proved unreliable in the iOS home-screen PWA (insets stayed 0, tab bar sat under the home indicator), so the hook takes `Math.max` of the context value and a direct DOM measurement of `env(safe-area-inset-bottom)`, with a 34px fallback for home-indicator iPhones in standalone when env() itself reports 0. Used by the tab bar (`app/(tabs)/_layout.tsx`), `AddMenu`, and the toast. It tolerates a missing SafeAreaProvider (the root toast overlay has none).
