# CLAUDE.md — Ours (private couples app)

Context for anyone (human or model) picking up this codebase.

## What this is

**Ours** is a private, just-for-two relationship app. One Expo codebase runs on iOS, Android, and web. Original brief in `Details.md`; it has since grown: a public landing page, a post-login home dashboard, a notification service, a calendar-based memory log, heart reactions, a shared bucket list, optional pairing, and (phase 2) a daily prompt with mutual reveal, time capsules, a date planner, per-partner wishlists with secret gift plans, and Sunday weekly reflections. **No stubs, no mock states.**

## Stack

| Layer | Tech |
|---|---|
| App | Expo SDK 53 + React Native 0.79 + Expo Router 5 (`react-native-web` for web) |
| API | ONE Vercel serverless function (`api/index.ts`) routing to modules in `api/_routes/` |
| Database | CockroachDB (Postgres wire protocol, `pg` driver) |
| Realtime | Ably (server publishes via REST, clients subscribe via scoped tokens) |
| Auth | Email+password, `node:crypto` scrypt, `jsonwebtoken` JWTs (30-day) |

## Commands

```sh
npm install
npm run migrate        # applies db/schema.sql (idempotent); RE-RUN after schema changes
npm run typecheck      # tsc --noEmit over app/ src/ api/ scripts/
npm run build:web      # expo export --platform web -> dist/
npx vercel dev         # API on :3000 (local)
npx expo start         # app; press w for web
```

Environment: `.env` from `.env.example` (`DATABASE_URL`, `ABLY_API_KEY`, `JWT_SECRET`, `EXPO_PUBLIC_API_URL` (stays EMPTY on Vercel), `VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY`/`VAPID_SUBJECT` for Web Push (generated once via `npx tsx scripts/generate-vapid.ts`), `MASTER_ENCRYPTION_KEY` for envelope encryption at rest (256-bit, base64; generated once via `npx tsx scripts/generate-master-key.ts`; if unset the app runs with encryption OFF so nothing breaks), `ADMIN_PASSWORD` gating `/admin/dashboard` (unset = dashboard locked), and `CRON_SECRET` gating the scheduled reminder endpoint `/api/cron/reminders` that the GitHub Actions in `.github/workflows/` call (unset = reminders return 503; the Actions also need repo secrets `CRON_SECRET` (same value) + `APP_URL`), and `AXIOM_TOKEN`/`AXIOM_DATASET` for searchable logs (unset = structured logs still go to stdout, see Observability)). Deployed via GitHub import into Vercel; env vars live in Vercel project settings.

`npm test` runs vitest over `api/**/*.test.ts` AND `src/**/*.test.ts` (server route unit tests plus the app's PURE helpers, which must stay free of React/react-native imports to run under plain node; `memory-item.test.ts` guards the per-user hearts rule, `onboardingSteps.test.ts` guards the first-run step rules).

## Product structure

- **Landing** `app/(auth)/welcome.tsx`: public marketing page (hero, three feature cards, CTA). Signed-out users at `/` land here.
- **Auth** sign-up/sign-in. **Signup auto-creates a solo space** (couple row); pairing is optional.
- **Pairing** `app/pair.tsx`: NOT a gate. Share your code or join theirs; joining migrates everything you authored into their space (see `api/_routes/couple-join.ts`).
- **Home** `app/(tabs)/index.tsx`: days-together count (basis: earliest anniversary milestone, else couple created_at; when there is NO anniversary yet the hero shows "Here is to you two ♥" + an inline "add the day it began" link instead of a misleading "0 days"), hero marks (both partners' avatars with a ♥ between; falls back to the initials monogram), invite banner when solo, "Your journey" points/level card, resurfaced memory (1 year ago today > 1 month ago today > random older), "Coming up soon" milestones (always rendered, with a "+ Add" header link; empty-state copy when none), latest pinned note, resurfaced memory. The bucket list is NO LONGER on Home (moved into the Wishes tab). Powered by ONE aggregate call: `GET /api/home` (still returns `bucket` for now, unused by the client).
- **Wishes** `app/(tabs)/wishlist.tsx` (ONE tab, route `/wishlist`, label "Wishes"): the third redesign of this surface and the one the user chose from pitched options ("Wishing on stars"); the history in one line: separate screens -> merged with Ours/Mine/Theirs segments (hated) -> six tabs with a separate Bucket list tab (hated: "I don't want it to move to a separate tab") -> THIS. Two halves split like Memories splits calendar/timeline: side by side on web >= 900px, one scroll on narrow, nothing hidden behind segments. (1) **The night sky** ("Wished together", data = the old `bucket_items`, unchanged): an oxblood panel where every open shared wish is a gold ✦ (deterministic scatter by id hash, decorative `SKY_DUST` glyphs); serif "We wish..." composer + gold ✦ send; tapping a wish opens inline actions *It came true ♥* (marks done + fires `showHearts()`), *Plan it as a date* (navigates to `/dates` with `{ compose, wish: title }`, the propose sheet prefills), *Rewrite*, *Let it go* (two-step). **"Pull one down"** draws a RANDOM open wish into a Sheet ("This one fell for you") with plan-as-date / pull-another. Come-true wishes leave the sky for a parchment **"Came true"** list (gold ✦ + date, tap = back-to-sky/remove). (2) **Gift wishes**: Mine/Theirs mini pill toggle; tag-styled rows where the tag's "hole" doubles as the got-it tick on the partner's list; secret plans carry a small wax dot + "only you see this"; composer keeps the secret-vs-visible choice. **The experience|item category taxonomy is DEAD in the UI** (user: "why does it have to be experiences and things"); the columns remain, the server defaults them, clients never send or show them. FAB action is "Make a wish" (focuses the sky composer). Wishlist title/url/notes stay encrypted; bucket titles plaintext.
- **Memories** `app/(tabs)/memories.tsx`: two parts. (1) Month calendar; days that hold a memory render a ♥ instead of the number; tapping a (non-future) day opens the composer pinned to that date (`memory_date` column). (2) Photo timeline with heart reactions and a full-photo viewer. Tapping the photo or note opens the viewer; the MessageCircle button (always visible) expands the comment thread INLINE under the card, Facebook-style (`MemoryComments` with `variant="light"`; the dark variant stays in the viewer). Comments can be hearted (v16 `comment_hearts`): partner-only like note hearts (never your own, server-enforced), ♡/♥ toggle on their comments, a quiet "♥ Loved" on yours, settled counts over `comment.hearted`. On web >= 900px calendar and timeline sit side by side. `MemoryImage` falls back to fetching the full photo for rows created before thumbnails existed (thumb_data null but has_photo true); do not remove that fallback.
- **Notes** `app/(tabs)/notes.tsx`, the letter wall (redesigned): composer card at TOP (serif input whose placeholder is a rotating "spark" starter; ✦ Sparkles button deals another, Smile toggles the emoji palette, Lock seals a capsule, wax-seal ♥ disc sends; no helper copy under it, just the LockBadge). Wall cards carry a tiny deterministic tilt (`tiltFor(id)`, pinned cards sit straight on blush), and ONE footer row: avatar + name + relative time on the left, icon tools on the right (Pin icon, gold + filled when pinned; Trash2 on your own; then the heart). Hearts: you can heart only your partner's notes; your own shows a bare "♥ Loved" once they do (`note_hearts` v9, PATCH /api/notes/:id { hearted }, settles on the server response). Kept deliberately word-light per user feedback: icons over labels, no explanatory text.
- **Avatars ("marks")** `src/components/Avatar.tsx`: curated set of lucide icons on tinted discs (ids must match `AVATAR_IDS` in `api/_routes/auth-profile.ts`); no upload, no emoji. Picked in Settings → Profile ("Your mark"), stored on `users.avatar` (v9), carried ONLY by `/api/auth/me` (self + partner) — list routes never join it because every author in a couple is one of the two `useAuth()` people. Fallback = serif initial disc. Shown on the Home hero, note cards, comment threads, and the Settings partner row.
- **Partner nicknames** (`users.partner_nickname` v11): a pet name you give your partner. Stored on the VIEWER's own row ("the name I call my partner"), so each partner sets their own independently. Resolved at `/api/auth/me`: `partner.display_name` becomes `nickname || realName`, so every screen reading `useAuth().partner` shows it with no other change; `partner.realName` + `partner.nickname` ride along for Settings and the notification-text swap. Set in Settings → Your space (calls `updateProfile({ partnerNickname })` then `refresh()`). The two server-generated name surfaces are handled explicitly: the notifications screen swaps `realName`→`nickname` in each stored line (`withNickname`), and `home.ts` resolves the pinned-note author. Plaintext by design (a term of endearment). Nullable/blank clears it. Catch-guarded selects (auth-me + home) keep a pre-v11 deploy from failing.
- **Onboarding** `app/onboarding.tsx` (route OUTSIDE the tabs group, v17): the guided first run, shown once to a NEW signup and never to an existing account. Gated by `users.needs_onboarding` (v17, `NOT NULL DEFAULT false` so every existing row is already "done" without a backfill UPDATE, which CockroachDB would reject in the same schema.sql; `auth-signup.ts` is the only place that sets it true, `PATCH /api/auth/profile { onboarded: true }` clears it). `/api/auth/me` carries `needsOnboarding` (catch-guarded); the tabs layout redirects on it. ONE route holding a step machine, not five routes. Steps: **add to home screen** (first, see its bullet for why) → **partner invite** (share a link / join theirs, reuses `joinSpace`) → **anniversary** → **your birthday** → **partner nickname** → **notifications**. Every step is skippable and leaves the field as empty as before; nothing is a gate (pairing especially). Which steps appear is decided by the pure, unit-tested rules in `src/lib/onboardingSteps.ts`: a step drops out when its data already exists (so closing the app mid-flow does not re-ask), the nickname step only exists once paired, and the notifications step only on a platform that can actually deliver. NOTE: there is no separate "your name" step, the signup form already collects it; the flow greets them with it instead. **Two first-run bugs found on device, do not regress**: (1) `signUp()` did not call `refresh()` the way `signIn()` does, so the space created during signup was absent from context and step 1 showed an EMPTY invite code until something else happened to hit `/auth/me`; (2) the notifications step never appeared on an iPhone because `canNotify` required `webPushSupported()`, which is false in a Safari tab (iOS exposes PushManager only to the installed app), so the step was filtered out instead of telling them to install. `canNotify` is now `webPushSupported() || webPushNeedsInstall()`, and the step renders an "Add Ours to your home screen" variant. `webPushNeedsInstall()` lives in `push-web.ts` and is shared with `NotificationsInvite`. (3) It STILL did not appear, because `ensureWebPushSubscribed()` in the tabs layout runs even on the render that redirects to onboarding (hooks cannot be conditional): on a device where permission was already granted it subscribed the brand new account within milliseconds, and the step then skipped itself as "already done". Two fixes: that effect is now gated on `!needsOnboarding` (onboarding owns the notification story for new signups), and **an existing subscription no longer removes the step at all** (`hasPushSubscription` is gone from `StepContext`); the step always shows on a capable platform and picks its own wording, including an "Notifications are on" confirmation variant.
- **Add to home screen** (`src/components/AddToHomeScreen.tsx` + `src/lib/install.ts`): the FIRST onboarding step, on **both platforms**, for a reason worth not forgetting. On iOS the home-screen app gets a **completely separate storage jar** from the browser (cookies, localStorage, even the service worker instance are not shared), so installing always costs one sign-in. Asking at the END of the flow made people redo a journey they had just finished; asking first costs only that sign-in, and everything else lives server-side (`needs_onboarding`, milestones, the couple) so the flow RESUMES. Do not move this step later. The screen is a DRAWN guide: a little phone built from views with the button you need circled in gold and an arrow pointing at it, because "tap share" is useless if you do not know where share lives. **Every iOS browser can install from the share sheet since iOS 16.4** (Safari, Chrome, Edge, Firefox, Orion, all WebKit, all giving the same push-capable home-screen app) so NOBODY is sent to Safari, an earlier version did that and it was wrong; the only per-browser difference is where the share button sits, which the arrow handles (Safari bottom toolbar, other iOS browsers top). On Chromium we capture `beforeinstallprompt` (`captureInstallPrompt()` runs early in the root layout, it fires once soon after load) and offer a REAL one-tap install. `shouldOfferInstall()` = any uninstalled phone; desktop never.
- **Session hand-off across the install boundary** (`src/lib/handoff.ts`): because of that separate storage jar, installing would otherwise dump someone on the sign-in screen seconds after signing up. Cache Storage is the one thing reported to be shared between the two iOS contexts, so `applySession()` stashes the token there and the installed app's first launch claims it (`claimSession()` in the auth bootstrap, before deciding they are signed out). Treated as a HAND-OFF, not storage: 15-minute expiry, deleted the moment it is read, cleared on sign-out. It is a documented trick rather than a supported API, so every call is wrapped and the fallback is simply the sign-in screen, where iOS Keychain autofills (both auth screens already carry the right `autoComplete` attributes, keep them).
- **Sharing the invite** (`src/lib/share.ts`): the code is the part people give up on, so the primary action on both the onboarding partner step and `app/pair.tsx` is **Send the invite**, which opens the phone's own share sheet (`navigator.share`, falling back to the clipboard) with a message plus `/sign-up?join=CODE`. The person who taps that link **never sees a code**: sign-up captures `join` and calls `joinSpace()` straight after the account exists (best effort, a stale code must not strand them). If they already have an account, the `(auth)` layout carries the code to `/pair?join=CODE` instead of bouncing them home and losing it. Reading the code aloud still works, demoted to a secondary affordance.
- **Milestones** yearly recurrence for anniversary/birthday, second-level countdowns. `milestones.person_id` (v17, nullable) says WHOSE a milestone is, which only birthdays need; anniversaries and every pre-v17 row leave it null and readers fall back to `author_id`, then the title text. `POST /api/milestones` accepts `personId` and validates it is a member of the couple; both the list and insert fall back to the pre-v17 column list so a deploy before the migration keeps working.
- **Daily prompt** (`api/_routes/prompts.ts`, home card + `app/(tabs)/prompts.tsx` history): one static-pool question per day, deterministic by date; answers are private until BOTH partners submit, then reveal. One submission per person per day.
- **Streak** (streak math in `api/_lib/streak.ts`, wired in `prompts.ts`): a day counts when BOTH partners answered that day's prompt. **Computed from the source of truth** (`computeStreak(mutualDays, todayUTC)`): the whole streak is re-derived every read from the set of days both partners answered, so it can never drift (the old incremental counter double-fired / mis-timezoned and showed wrong numbers). Pure + unit-tested (`api/_lib/streak.test.ts`, 14 cases incl. "2 days reads as 2" and the grace-visibility set). UTC days, one grace day per Monday-to-Sunday UTC week; `computeStreak` also returns `graceAvailable` + `graceDay` so the UI can say plainly whether the week's one free miss is still in hand. `couples.current_streak_days/longest_streak_days/last_streak_date` are now just a cache for admin analytics, refreshed on each reveal by `advanceStreak`; `grace_used_week` is unused. `streakStateFor` = read view. Surfaced from day one: hero chip on Home, prompt-card lines, and the prompts screen header carries a persistent "how the streak works" explainer (user could not tell what it counted). **The full streak screen** `app/streak.tsx` (outside the tabs group, opened by the Home chip and a "See your streak" link on the prompts screen): the count + what it means today, a **grace day card** (unused / used, naming the day it covered; free and automatic, and deliberately NOT a purchasable freeze since Ours has no billing), both partners side by side with today's state, and a **shared month calendar** mirroring the Memories one (♥ = both answered and the day counts, ♡ = only one of you, today outlined). Fed by `GET /api/prompt/today?days=1`, which adds a `days: [{date, mine, theirs}]` array (one year, one grouped query, no answer text read); the flag keeps the plain prompt card a light request. **Streak celebration**: when a day becomes mutual (both answered), the sanctioned hearts shower fires once per prompt day as positive reinforcement (Home `celebrateStreak`, guarded by `celebratedDateRef`; fires from the answer-submit response for the second answerer and from the `prompt.revealed` event for the first). No leaderboards.
- **The winding path** (in `home.ts`; was "story chapters", then "relationship points/levels"): the second retention hook, now on its THIRD framing, chosen by the user from pitched options after "too technical" and "no xp should be there". **Numbers are never shown to the user, anywhere.** The Duolingo lesson applied: progress reads as a PLACE on a road, the score is internal plumbing. Server side nothing moved: points math stays in home.ts (`POINTS`: memories/dates_done x5, answers/bucket_done x3, notes/milestones x2, guesses x2 (v16, catch-guarded), comments x1) with `LEVELS` thresholds naming the places (First Glance -> Getting Closer -> Finding Our Rhythm -> Love Letters -> **Slow Dances** (renamed from "Keepsakes", which the user flagged for its negative connotation; do not reuse that word in copy) -> Golden Hours -> Building a Life -> The Long Song -> A Thousand Days -> Ever After); computed on read, cannot drift; response `story: { points, level, levelTitle, levelStart, nextAt, counts, recent }` unchanged. Client "Your path" card: a dotted road of 10 nodes (passed = gold, current = ♥, ahead hollow), "You two are at {place}" + "The road bends on toward {next}"; TAP expands a plain-language line + "What moved you lately" (`story.recent` rendered with NO point values, only `RECENT_LABEL` phrases + dates) + "See the whole road" (Sheet: places marked passed-through/you-are-here, zero numbers). Reaching a new place = the sealed "Somewhere new" ceremony with `madeOfSummary` ("A place made of 18 memories, 12 answered questions and 9 notes", counts of real things, never points); last celebrated level still in `localStorage['ours.story-chapter']`. LEVELS is duplicated client-side; keep both in sync. Reaching a level shows a one-time sealed "A new level" ceremony with a top-3 `pointsSummary`; last celebrated level in `localStorage['ours.story-chapter']` (key kept; web only, native skips). LEVELS + POINT_SOURCES are duplicated client-side in `app/(tabs)/index.tsx`; keep both in sync with home.ts.
- **Chat** (`app/chat.tsx`, `api/_routes/messages.ts`, v10): a direct partner-to-partner thread. Reached via a round **chat button parked above the add FAB** (`src/components/ChatButton.tsx`, rendered once in the tabs layout; shown only when paired + on the main tab routes; hides while the add menu is open via the `src/lib/fabMenu.ts` shared signal; carries an unread dot). Bodies are **encrypted at rest** (envelope, `messages.body_ct`). Live over Ably (`message.created`, plaintext body over the TLS + scoped channel, like notes/memories) + a best-effort generic Web Push to the away partner. Chat deliberately writes **no notification rows** (would flood the bell); unread is tracked by `users.chat_seen_at` (`GET /api/messages/unread` for the badge, `POST /api/messages/seen` to clear). Inverted FlatList, optimistic send, `?before=` pagination. **Media (v15)**: a message can carry a photo (`image_thumb` ~480px shown inline + in the list, `image_data` full fetched on tap via `GET /api/messages/:id`; plaintext base64 like memory photos, NOT encrypted). The thumb rides the `message.created` event (list-weight, deliberate minor deviation from "no images over Ably"). An image bubble has an **"Add to timeline"** button (`POST /api/messages/:id { action:'to-timeline' }` server-side copies the photo into a `memories` row, which DOES notify). **Last seen**: `/messages/seen` publishes `chat.seen`; the sender sees a "Seen" receipt under their newest message once the partner's `chat_seen_at` passes it; GET returns `partnerSeenAt`. **Replies (v16)**: long-press a bubble to quote it (`messages.reply_to_id`, id only; the client renders the quote from its loaded thread, "An earlier message" when it is beyond the page). **Text rendering**: `LinkedText` linkifies http(s)/www URLs (tap opens, trailing punctuation excluded), and the bubble chain carries `flexShrink` + web `overflowWrap:'anywhere'` so long unbroken tokens WRAP instead of pushing the bubble off screen (that was a real reported bug; do not remove).
- **Hearts shower** `src/components/HeartsRain.tsx` (mounted once in the tabs layout next to NudgeToast): a ~2.5s rain of ♥ glyphs (oxblood/gold/blush, staggered falls, pointerEvents none). Fires live on the `nudge` Ably event (not your own), on app open via `showHearts()` when `/api/home` returns `nudged: true` (an unseen nudge from the last 48h, computed with a FILTER clause on the unseen-notifications query; Home guards it to once per open), and now on a **streak extension** (a prompt day becoming mutual, once per day). It is the ONE sanctioned shower; reuse `showHearts()` for celebrations, do not add a second confetti system.
- **Time capsules**: notes and memories accept `sealedUntil`. Partner-authored sealed rows come back with content stripped server-side (never client-side). After the reveal date they show as "ready to open"; first open is recorded in `capsule_opened_at` and notifies the author. Memory opens happen via GET /api/memories/:id; note opens via PATCH /api/notes/:id { open: true }.
- **Date planner** `app/(tabs)/dates.tsx` (rebuilt): three segments **Upcoming** (accepted, not-yet-happened) / **Proposals** (open, waiting on you/them) / **Past** (happened or resolved). Flow: propose title/location + optional date + optional **time** (HH:MM, `proposed_time` v13) → the OTHER partner accepts/declines/counters (PATCH /api/dates/:id { action }, counter carries time too); accepting a dated proposal still creates a custom milestone in the same transaction. Confirmed dates land in Upcoming; **reminders** fire ~24h/6h/1h before via the cron (see Reminders). Once a date has passed, its Past row invites **"Log how it went"** (`action:'complete'`): a `ReflectSheet` with a 1-5 **star rating**, a **note**, an optional **photo that becomes a real timeline memory** (the client POSTs /api/memories then passes `memoryId`), and a **"save as a date idea"** toggle (writes `date_ideas`, encrypted title/location). `completed_at`/`rating`/`reflection`(_ct)/`memory_id` on the proposal (v13). GET /api/dates also returns `ideas`; the propose sheet's **"Surprise them"** picks a random idea from the couple's saved `date_ideas` ∪ a built-in list (`BUILT_IN_IDEAS`). Every row still opens a status-aware `ProposalDetail`, which also carries the **manage tools**: *Change it* (opens the propose sheet in edit mode, `PATCH { action:'edit' }`) and *Call it off / Remove* (two-step, `DELETE /api/dates/:id`). **Ownership rule, enforced server-side in `canModify`**: an OPEN proposal is still just an idea and belongs to whoever proposed it; an ACCEPTED date is a shared plan, so either partner may edit or cancel it (a deliberate second exception to own-content-only deletes, alongside memories). A logged date cannot be edited. Editing keeps the acceptance's milestone in step (updates it, deletes it when the day is cleared, creates one when a day is added to an accepted date) and **resets `reminded_24/6/1` whenever the day moves**, or the new schedule would be silenced by the old flags. Accepted dates also offer **"We already did this"**, since `complete` was never gated on the day having passed: a date can happen early, or have no day at all. Date notifications + reminders are deliberately generic (title is encrypted). The propose sheet also opens prefilled from a Wishes night-sky wish (`/dates?compose=...&wish=<title>`).
- **Wishlist** `app/(tabs)/wishlist.tsx`: each partner owns a list, the other reads it. `secret` rows are gift plans added to the PARTNER's list, hidden from the owner forever (server-side filter, do not weaken). Partner toggles `gotten`; owner never sees who got it.
- **Weekly reflection / week-in-review** (`api/_routes/reflection.ts`): Monday-to-Sunday UTC counts computed on read, home card on Sundays only, savable keepsakes browsable at `app/(tabs)/reflections.tsx`. `computeReflection` now also gathers a **snapshot**: a few of the week's memory thumbnails with their (decrypted) notes (`gallery`) + a couple of decrypted note excerpts (`noteHighlights`); the Home card and history render a little photo strip + a note quote above the counts. Saving **freezes** the snapshot into `weekly_reflections.snapshot` (JSONB, v14) so the keepsake survives later edits (ON CONFLICT now upserts the snapshot). NOTE: this also fixed a latent bug where the highlight note was returned as raw (empty under encryption) instead of decrypted.
- **This-or-That daily game** (`api/_routes/game.ts`, v16 `daily_game_answers`, Home card "Today's This or That"): the third retention hook. One pair of options per day (static `GAME_POOL`, deterministic by date like prompts); each partner picks their own answer AND guesses their partner's, two taps. Mutual reveal (nothing shows until both play, picks reduced to a boolean server-side before then). **Styled as the prompt card's sibling** (user hated the first plain version): unplayed = SEALED oxblood card, serif question, two parchment option buttons with a gold italic "or" between; reveal = light card with both partners' avatar marks facing each other over their picks, ♥ between, and a serif verdict line; no points ever mentioned in its copy (right guesses feed the path silently, x2, counted in home.ts from `a.guess = b.pick` joins). `GET/POST /api/game/today` (`{ pick, guess }`, 'a'|'b', 409 on replay via ON CONFLICT DO NOTHING); state rides `/api/home` (`game`). Ably `game.updated` is id-free (never carries picks); the reveal notifies the FIRST answerer (kind `game`). Only letters land in the DB, nothing to encrypt.
- **Friend referral** (`api/_routes/referral.ts`, v16 `users.referral_code`/`referred_by`, Settings → "Share Ours"): a personal share link, `/sign-up?ref=CODE` (8-char invite alphabet, minted lazily by `GET /api/referral`, which also returns `joined` = count of users whose `referred_by` is you). Signup accepts `ref` (best-effort, catch-guarded, never blocks). Deliberately NO reward mechanics: relationship points stay about the couple.
- **Notifications** `app/(tabs)/notifications.tsx` (hidden tab, reached via bell): every partner action (kinds: nudge, memory, note, milestone, partner, bucket, prompt, capsule, date, wishlist, comment, game), stored in `notifications` AND pushed live over Ably. Unread dot in `src/lib/notifications.tsx`; `users.notifications_seen_at` tracks read state.
- **Navigation**: 5 tabs (Home, Memories, Notes, Dates, Wishes); Milestones, Settings, Notifications, Prompts, Reflections are hidden tabs (`href: null`) reached from home/bell/TopNav. Web >= 900px gets `src/components/TopNav.tsx`; native/narrow gets bottom tabs + `src/components/HeaderActions.tsx`. Home has no header; its hero row carries Nudge/Bell/Settings. Bottom-right controls, stacked: the chat button sits above the wax-seal add FAB (five actions: memory, note, make a wish, date, milestone).
- **Admin dashboard** `app/admin/dashboard.tsx` (route `/admin/dashboard`, OUTSIDE the auth/tabs groups): counts-only analytics. Overview (couple/user counts, paired vs solo, encrypted couples, active-this-week), streaks, 30-day signups chart, **content magnitude bars incl. chat messages**, a **30-day activity chart with a Daily/Cumulative toggle** (`activity` series, all content types), and a **per-couple breakdown** (`couples`: short opaque id `left(id,8)`, member count, total items + a per-type line + joined/last-active dates, sorted by volume). Gated by one shared password in the `ADMIN_PASSWORD` env var: `POST /api/admin/auth` returns a 12h admin JWT (`{ admin: true }`, no `sub`, kept separate from user auth in `api/_lib/admin.ts`), `GET /api/admin/stats` requires it. Reads only COUNTS + timestamps (per-couple counts were added at the owner's request); never reads couple content, never decrypts. The admin token lives only in the screen's state, never localStorage, never the shared `api()` client. If `ADMIN_PASSWORD` is unset the dashboard stays locked (503).

### Envelope encryption at rest (`api/_lib/envelope.ts`)

The single encrypt/decrypt boundary (a future move to E2E is a swap of this module + a client layer, not a route rewrite). Two key layers: a **master key** (256-bit, base64 in `MASTER_ENCRYPTION_KEY`, never in the DB) wraps a per-couple **DEK** (256-bit random, stored AES-256-GCM-wrapped on `couples.wrapped_dek`); field values are encrypted with the couple's DEK. Ciphertext layout per value: `iv(12) || ciphertext || authTag(16)` as one `BYTEA`.

- API: `encryptionEnabled()`, `freshWrappedDek()` (used by `invite.createCoupleForUser`), `encryptField(coupleId, plaintext) -> Buffer|null`, `decryptField(coupleId, blob) -> string|null`, `readField(coupleId, ct, plaintext)` (prefer decrypted ct, else the plaintext column).
- **Graceful degradation**: with no master key set, every helper returns null and callers read/write plaintext exactly as before, so the deployed app keeps working until the key is provisioned. Legacy couples get a DEK minted lazily on first use.
- **Storage pattern**: each encrypted column has a `<field>_ct BYTEA` beside the original. When encryption is ON, writes store the plaintext column empty (`''`/NULL) and the ciphertext in `_ct`; reads use `readField` (decrypt `_ct`, fall back to plaintext for old/never-backfilled rows). Old plaintext columns are NOT dropped (additive only). `scripts/backfill-encryption.ts` encrypts pre-existing rows (idempotent; blanks plaintext after).
- **Encrypted fields**: `memories.note`, `love_notes.body` (covers note/memory time-capsule content), `daily_prompt_answers.text`, `wishlist_items.title/url/notes`, `date_proposals.title/location`, `messages.body` (chat). NOT encrypted (by design): milestone titles (list-view), bucket titles, reflection counts, timestamps, FKs, wishlist `secret` flag. `notify()` text must never embed an encrypted value (it would leak into the plaintext `notifications` table) — wishlist/date notifications are deliberately generic.
- **Pairing migration**: `couple-join.ts` re-wraps the migrated encrypted fields (`memories.note_ct`, `love_notes.body_ct`) from the joining user's DEK to the target couple's DEK inside the merge transaction (`recryptBlob`), so content stays readable under the new key. Any encrypted field that becomes migratable later must be added to that route's `MIGRATED_CT` list.
- **Ably** carries plaintext for these fields (server already holds it; the channel is TLS + subscribe-only scoped tokens). "Encryption at rest" is about the database, not the realtime hop.
- **`/api/auth/me`** returns `encryption: boolean` plus `encryptionCode` (the "seal code": `keyFingerprint()` = SHA-256 of the couple's DEK with a fixed context string, mapped to 8 chars of the invite alphabet as `XXXX-XXXX`; one-way, identical for both partners, shown in Settings → Privacy so the couple can compare phones). The client threads both through `useAuth()`; `LockBadge` on composers and the Privacy copy never overclaim when the key is absent.
- **Key rotation (not automated)**: to rotate `MASTER_ENCRYPTION_KEY`, for every couple: unwrap `wrapped_dek` with the OLD master key, re-wrap with the NEW one, write it back (one pass, DEKs and field ciphertext are untouched). Rotating a couple's DEK instead means decrypting every field with the old DEK and re-encrypting with the new one. Never change the master key without re-wrapping, or all data becomes unreadable.

## Observability (`api/_lib/log.ts`, `src/lib/log.ts`, `POST /api/logs`)

One structured-JSON pipeline, server and client. Every line is `{ _time, level, event, service, env, ...fields }`.

- **Sinks**: always stdout (Vercel function logs); additionally **Axiom** when `AXIOM_TOKEN` + `AXIOM_DATASET` are set (`AXIOM_URL` only for the EU region). Unset = stdout only, nothing breaks.
- **Server**: `log(level, event, fields)` buffers per invocation; `route()` flushes ONCE **after** the response is written, so shipping logs never sits in the user's latency path. `flushLogs()` never throws and gives up after 2.5s.
- **Automatic coverage**: `route()` emits `http.request` (method, path, status, duration_ms, requestId, plus the error message on a 4xx/5xx) for every endpoint, so a new route is instrumented the moment it exists; `requireUser` stamps `user_id`/`couple_id` onto the request's log context (`logContext(req, {...})` adds more). `api/index.ts` logs `http.no_route`. `db.ts` logs `db.query_failed` + `db.slow_query` (>1s).
- **Push has the deepest instrumentation** because it is the least observable feature: `push.send` on EVERY attempt with a machine-readable `reason` from `PushReason` (`notifications-off` | `no-subscription` | `vapid-not-configured` | `invalid-subscription-json` | `subscription-expired` | `send-failed` | `native-not-provisioned`), plus `endpoint_host`, the push service's own `push_status`/`push_body`, and a `source` tag naming the caller (`notify:<kind>`, `cron:daily`, `cron:dates`, `chat`, `test`). Also `push.vapid_missing`, `push.subscription_stored`, `notify.sent`.
- **Cron reminders**: `POST /api/cron/reminders` returns a full report (`users`, `eligible`, `sent`, `skipped{reason:n}`, `failed{reason:n}`, `missing_vapid_env`) and logs it as `cron.summary`; `cron.date_checked` carries each date's `hours_until`. The GitHub workflow PRINTS the body, fails the job on a non-2xx, and warns on `"sent":0`, so a green run that delivered nothing says why in the job log. `cron.unauthorized` catches the classic repo-secret-vs-Vercel-env drift.
- **Client**: `logEvent(event, fields)` buffers and POSTs batches to `/api/logs` (3s debounce, 20 per batch, flush on tab hide). `installGlobalLogging()` (called once in `app/_layout.tsx`) catches uncaught errors and unhandled rejections; `api.ts` logs `client.api_failed`; `push-web.ts` logs the whole subscribe path (`push.sw_registered` carries `permission` + `subscribed`, which is usually the answer). Intake is auth-OPTIONAL (signed-out failures matter most), caps 50 events, and namespaces client fields as `c_*` so they can never forge `user_id`.
- **Installing must never cost someone the notification ask (do not regress)**: the ask schedule lives in `src/lib/pushAsk.ts` and is capped, but `askIsDue(record, standalone)` treats **becoming installed as earning one fresh ask**, overriding both the cooldown and the three-ask cap. Reason: every earlier "maybe later" was answered in a browser tab, which on an iPhone could not have subscribed even if they had said yes, and on Android the installed app SHARES localStorage with the tab, so the cooldown would otherwise follow them in and meet the person who installed specifically to get notifications with silence. `askedStandalone` on the record is what makes it fire only once. Only `done` (granted, or blocked at the browser level) still stops it. Covered by `src/lib/pushAsk.test.ts`. (On iOS the point is partly moot because the home-screen app gets its own storage jar, so the record starts empty there anyway, and they will need to sign in again.)
- **The notification invitation** (`src/components/NotificationsInvite.tsx`, mounted once in the tabs layout): a warm centered modal (wax-seal ♥ disc, serif copy) that asks BEFORE the browser's own prompt is ever fired. This is permission priming, and it is deliberate: a browser gives exactly one cold prompt and a dismissal is permanent, so only people who tap "Yes, keep me close" ever see the real one. Shows when signed in + on web + the server holds no subscription; three copy variants (`paired` naming the partner, `solo`, and `install` telling iPhone-in-a-tab users to add Ours to the home screen first, since Safari cannot subscribe otherwise). **Cadence is capped on purpose**: `localStorage['ours.push-ask']` = `{n, at, pairedAsked, done}`, spaced today / +3 days / +10 days, then never again; a granted or browser-blocked answer sets `done`. The `partner.joined` event gets ONE extra ask outside the schedule (the most persuasive moment there is). Do not make this fire every open: nagging is what converts "maybe later" into a permanent browser-level block.
- **The switch must show DELIVERY, not intent (real bug, do not regress)**: `users.notifications_enabled` is `NOT NULL DEFAULT true`, so binding the Settings Switch to it alone showed "on" to every new account while `push_token` stayed NULL, and `enableWebPush()` only ever runs when someone FLIPS the switch. Nobody flipped an already-on switch, so nobody subscribed (a `kind=prompt` cron run reported `no-subscription` for 4 of 8 users). Now: on web the switch reads `notificationsEnabled && hasSubscription` from `GET /api/push/subscribe`, and `ensureWebPushSubscribed()` (`src/lib/push-web.ts`, called from the tabs layout on every signed-in load) silently re-subscribes when permission is ALREADY granted but the server has no live subscription (rotated endpoints, 410-cleared rows). It never prompts; prompting stays the switch's job.
- **Self-diagnostics**: `GET /api/push/subscribe` reports `serverConfigured` / `missingEnv` / `notificationsEnabled` / `hasSubscription` / `endpointHost`; `POST /api/push/subscribe { test: true }` sends a real push to yourself and returns `{ delivered, reason }`. Settings → Notifications surfaces both as "Check they are working" with a plain-language line per reason.
- **PRIVACY (hard rule)**: never log couple content. No note/memory/message/prompt/wishlist text, no notification `text`, no emails, no tokens, no query params (they carry ciphertext and passwords), no full push endpoints (host only, it is a capability URL). Log ids, kinds, counts, reasons, durations. Strings are truncated to 400 chars as a backstop, not a licence.
- Tested in `api/_lib/log.test.ts` (redaction, batching, failure tolerance) and `api/_routes/logs.test.ts` (intake sanitizing + spoofing).
